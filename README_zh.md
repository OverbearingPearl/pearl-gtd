# Pearl-GTD: 为 Emacs org-mode 设计的完整 GTD 工作流系统

[English](README.md) | [中文](README_zh.md)

本包在 Emacs org-mode 中完整实现了 David Allen 的《搞定》（GTD）方法论，提供了一套全面工作流系统。它涵盖 GTD 的五个核心步骤（收集、厘清、组织、检视、执行），并完整支持六层视野框架、自然计划模式、每周回顾流程、情境分类系统、两分钟原则以及所有标准 GTD 清单（项目、下一步行动、等待、将来/可能、参考资料）。

系统还包含收件箱处理、项目规划、定期回顾、参考资料管理、触发清单、头脑风暴工具和组织原则，帮助用户实现从日常任务到人生目标的垂直与水平关注管理。

## GTD 理念
GTD 方法论建立在几个根本性的哲学理念之上，这些理念构成了整个系统的基础：

1. **心智如水**：通过将任务外化到可信赖的系统中，让大脑从记忆负担中解放出来，专注于创造性思考而非存储信息。
2. **结果导向**：任何任务都必须有明确的期望结果，模糊的目标无法驱动有效行动，清晰的定义是执行的前提。
3. **系统信任**：建立一个完整、可靠的外部系统，让用户能够完全信任它，从而减少心理焦虑和决策疲劳。
4. **情境驱动**：行动的选择应基于当前的情境、可用时间、精力和工具，而非单纯的任务重要性或紧急性。
5. **持续进化**：通过定期检视和调整，系统需要不断适应变化的环境和目标，保持其相关性和有效性。

### 五个步骤

GTD 工作流包含五个连续步骤，形成一个闭环系统：

```
      +----------------+
      |      收集      |
      |    (Capture)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      处理      |
      |    (Clarify)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      组织      |
      |   (Organize)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      回顾      |
      |    (Reflect)   |
      +--------+-------+
               |
               v
      +--------+-------+
      |      执行      |
      |    (Engage)    |
      +----------------+
```

#### 1. 收集 (Capture)
将所有任务、想法、承诺和待办事项从大脑和外部来源收集到“收件箱”中。目标是清空大脑，确保没有遗漏。

#### 2. 处理 (Clarify)
对收件箱中的每个项目进行逐一处理：
- 问自己：“这是什么？”
- 如果不是可执行的任务，则丢弃、归档为参考资料或标记为“将来/可能”
- 如果是可执行任务，则确定下一步行动和期望结果

#### 3. 组织 (Organize)
将处理后的项目放入相应的系统中：
- **下一步行动清单**：可立即执行的具体行动
- **项目清单**：需要多个步骤才能完成的任务
- **等待清单**：委托他人或等待外部事件的任务
- **将来/可能清单**：暂时不执行但未来可能考虑的任务
- **参考资料**：需要保存但不需行动的信息

#### 4. 回顾 (Reflect)
定期检视系统以保持其更新和有效：
- **每日回顾**：检查当日行动清单和日程
- **每周回顾**：全面更新所有清单，回顾目标和项目进展
- **每月/年度回顾**：检视更高层次的目标和愿景

#### 5. 执行 (Engage)
基于当前情境、时间、精力和优先级，从清单中选择最合适的行动来执行。这是实际完成工作的阶段。

这五个步骤形成一个持续循环：执行后产生新的任务需要收集，定期回顾确保系统有效，从而持续提升个人生产力。

### GTD 工作流程图

以下是改进版的 GTD 工作流程图，基于《搞定》书中提出的经典工作流程，展示了从收集到执行的完整决策路径：

```
                              +-----------------------+

                              |      (inbox.org)      |
                              +-----------+-----------+
                                          |

              +-----------+-------------+        +--------+---------+

                          |           | 完成后删除该记录 | |  (Context? Calendar? Delegated? Project?)  |
                          |           +-----+------------+ +----+----------+---------+------------+-----+
                          |                 |                   |          |         |            |
                          |                 |                   v          v         v            |
                          |                 |          +--------+-+ +------+---+ +---+--------+ +-+--------+
                          |                 |          | 添加情景 | | 添加日程 | | 添加委托人 | | 添加项目 |

         |                |             |   |                |            |             |             |

         |                |             |   |                |            |             |             |
         v                v             v   v                v            v             v             v

 |  参考资料清单 | | 将来也许清单  | | 垃圾桶  | |                         行动清单                           |
 |(reference.org)| | (someday.org) | | (Trash) | |                       (actions.org)                        |
 +---------------+ +---------------+ +---------+ +------------------------------------------------------------+
```

这个流程图清晰地展示了如何处理收件箱中的每个项目：
1. **收集**：所有任务进入收件箱。
2. **识别**：明确任务的含义。
   - 是否需要重新命名任务标题。
   - 是否需要补充一些详细内容备注。
3. **处理**：判断是否可执行。
   - 如果不可执行：分配至参考资料、将来也许清单或丢弃。
   - 如果可执行：判断是否能在2分钟内完成。
     - 如果是：立即执行并删除该记录。
     - 如果否：进一步检查是否涉及情境、固定日程、委托他人或属于项目，并相应添加标签属性（如 :CONTEXT:、SCHEDULED:、:DELEGATED: 或 :PROJECT:）。
4. **组织**：根据判断结果，将任务分配至相应清单，例如行动清单。
5. **执行**：基于情境选择合适的行动。

通过这个流程图，用户可以直观地理解如何使用本插件实现 GTD 工作流，确保每个任务都得到恰当的处理和归类。

## 执行（Engage）

当任务经过收集、处理和组织后，用户进入执行阶段。系统提供基于情境的智能过滤，帮助用户在合适的时间选择合适的行动。

### 情境过滤

执行视图支持多维度动态过滤：

- **情境标签（Context）**：@office, @home, @phone, @computer, @errands 等
- **时间可用**：根据当前可用时间筛选可完成任务（5分钟/15分钟/30分钟/1小时）
- **精力状态**：高/中/低精力匹配不同复杂度任务
- **优先级**：基于目标和项目重要性智能排序

### 执行视图

专门的执行界面显示：
- 当前情境下的下一步行动清单
- 已逾期任务高亮提醒
- 委托给他人的任务状态（等待清单）
- 今日已完成的任务统计

## 回顾（Review）

定期回顾是维持GTD系统可信度的关键机制。系统提供结构化的回顾流程，确保所有清单保持最新。

### 周期性回顾

**每日回顾（Daily Review）**
- 检查日程和已预约事项
- 浏览当日可执行的行动清单
- 快速清空当日收集的杂项

**每周回顾（Weekly Review）**
- 清空收件箱（确保收集阶段无遗漏）
- 回顾所有项目（确保每个项目有明确的下一步行动）
- 检查等待清单（跟进委托事项）
- 回顾将来/也许清单（评估是否激活某些项目）
- 检视目标（L3）和关注领域（L4）的进展

**每月/季度回顾（Monthly/Quarterly Review）**
- 检视愿景（L5）和人生意义（L6）的对齐情况
- 评估长期目标的达成进度
- 调整关注领域和项目组合

### 高级过滤回顾

支持复杂条件组合的临时过滤查询：

```
示例场景：
- 未委托给他人（非等待状态）
- 属于特定项目（Project A）
- 没有指定日程（无SCHEDULED）
- 创建时间超过30天
- 关联特定目标（L3）
```

### 回顾编辑模式

回顾过程中，系统提供临时编辑窗口直接操作原始org文件：
- 批量修改属性（如统一更改情境标签）
- 重新安排日程或截止日期
- 调整任务与项目的归属关系
- 修改视野层级指定

所有变更实时同步回原始文件，确保数据一致性。

## 六层视野（6 Horizons of Focus）

六层视野模型提供垂直维度的对齐管理，确保日常行动与人生最高目标保持一致。

### 视野层级结构

```
+------------------------------------------------+
| 第6层：人生意义 / Purpose                      |
| 你存在的根本理由，核心价值观和终极使命         |
| 时间跨度：一生                                 |
+------------------------------------------------+
           ↓ 支撑和体现
+------------------------------------------------+
| 第5层：愿景 / Vision                           |
| 未来3-5年的成功图景，长期期望的状态            |
| 时间跨度：3-5年                                |
+------------------------------------------------+
           ↓ 支撑和体现
+------------------------------------------------+
| 第4层：目标 / Goals                            |
| 未来1-2年内要达成的具体目标                    |
| 时间跨度：1-2年                                |
+------------------------------------------------+
           ↓ 支撑和体现
+------------------------------------------------+
| 第3层：关注领域 / Areas                        |
| 需要长期维护的生活/工作领域（健康、财务、家庭）|
| 时间跨度：1-3年                                |
+------------------------------------------------+
           ↓ 支撑和体现
+------------------------------------------------+
| 第2层：项目 / Projects                         |
| 需要多个步骤在一年内完成的成果                 |
| 时间跨度：数周至一年                           |
+------------------------------------------------+
           ↓ 支撑和体现
+------------------------------------------------+
| 第1层：行动 / Actions                          |
| 可以立即执行的下一步具体动作                   |
| 时间跨度：当前                                 |
+------------------------------------------------+
```

### 层级指定规则

**单行动（Single Actions）**
- 只能指定至 **第3层（Areas）**
- 用于将独立行动与中期目标对齐
- 示例："买牛奶"关联到"家庭健康维护"（L3 Areas）

**项目（Projects）**
- 可以指定 **第3、4、5、6层**
- 用于确保项目与更高层次的关注领域、愿景和人生意义保持一致
- 示例："家庭装修"项目可同时关联：
  - L3（Areas）：居住环境
  - L4（Goals）：提升生活品质
  - L5（Vision）：拥有温馨的家庭港湾
  - L6（Purpose）：为家人创造幸福

### 对齐检查

系统提供自动对齐检查，识别：
- **孤儿行动**：未关联到L3（Areas）的单行动
- **无归属项目**：未指定L3-L6层级的项目
- **断裂对齐**：项目与更高层级不一致的情况

## 自然计划法（Natural Planning Model）

当需要从零开始规划一个项目时，自然计划法提供结构化的思考框架。

### 五步骤流程

**1. 定义目的和原则（Purpose & Principles）**
- 为什么要做这个项目？
- 成功的标准是什么？
- 约束条件和指导原则？

**2. 展望结果（Outcome Visioning）**
- 成功时具体会是什么样子？
- 在脑海中描绘完成的图景

**3. 头脑风暴（Brainstorming）**
- 捕捉所有想法，不加评判
- 数量重于质量，延迟组织

**4. 组织整理（Organizing）**
- 将想法分类、排序、建立结构
- 识别关键组件和依赖关系

**5. 确定下一步行动（Next Actions）**
- 识别可以立即开始的物理动作
- 创建项目支持材料和行动清单

### 从空白创建项目

用户可在无任何现有任务的情况下启动自然计划：
1. 执行 `M-x pearl-gtd-planning-start`
2. 系统引导完成五个步骤的头脑风暴
3. 最终生成完整的L2项目（Projects）
4. 为项目指定L4-L6层级对齐
5. 自动将头脑风暴内容转化为项目支持材料

## 完整工作流

Pearl-GTD 整合所有功能形成统一的工作流系统：

```
收集 → 澄清 → 组织 → 回顾 → 执行
  ↓      ↓      ↓      ↓      ↓
Inbox  Rename  Actions Daily   Context
       Notes   Projects Weekly Filter
       L3      L4-L6   Monthly Views
              ↑              ↑
         自然计划法      六层视野对齐
         （创建项目）    （垂直管理）
```

通过水平流程（收集到执行）与垂直维度（六层视野）的结合，系统帮助用户既能在日常行动中保持高效，又能确保所有活动与人生最高目标保持一致。
